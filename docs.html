DOLE Payroll System

DOLE Regional Office 9 Payroll Management System

A Laravel-based payroll system for DOLE RO9 that handles regular payroll processing, special payroll types (newly hired, salary differential, NOSI/NOSA), travel expense vouchers (TEV), office orders, and government remittance reports.

Who Uses It

The system serves multiple roles with specific permissions:

  • payroll_officer — Creates and computes payroll batches, manages employees
  • hrmo — Manages employees, office orders, TEV requests, files liquidations
  • accountant — Certifies payroll funds, reviews TEV requests
  • ard / chief_admin_officer — Approves payroll and TEV requests
  • cashier — Releases payroll, handles TEV cash advances and liquidations
  • budget_officer — Monitors TEV submissions for reference

Main Features

  • Regular Payroll — Semi-monthly payroll (1st and 2nd cutoff) with full workflow: draft → compute → submit → certify → approve → lock
  • Special Payroll — Newly hired pro-rated payroll, salary differential, NOSI/NOSA
  • TEV Management — Travel expense vouchers with itinerary, approval workflow, cash advance tracking, and liquidation
  • Office Orders — Authorization documents for official travel
  • Employee Management — Employee records with HRIS API sync, deduction enrollments, promotion history
  • Government Remittances — GSIS, HDMF/Pag-IBIG, PHIC, SSS, and various loan remittances (LBP, CARESS, MASS, Provident Fund, BTR)
  • Reports — Payroll register, payslips, TEV reports, remittance exports
  • HRIS Integration — Employee SSO via JWT authentication, attendance data synchronization
  • Deduction Type CMS — Full content management system for deduction types
  • Signatory Management — Document signatory assignment and management

Local Setup

The project supports two setup methods: XAMPP (recommended for Windows) and Docker (edge case for PCs without PHP).

XAMPP Setup (Recommended)

Note: An automated setup script is available: setup-xampp.bat
  1. Install XAMPP from apachefriends.org
  2. Start Apache and MySQL services in XAMPP Control Panel
  3. Clone the repository
  4. Run setup-xampp.bat (this auto-detects XAMPP, creates database, runs migrations, imports data)
  5. Or manually:
    • Copy .env.example to .env
    • Run php artisan key:generate
    • Configure database in .env (XAMPP defaults: DB_HOST=127.0.0.1, DB_USERNAME=root, DB_PASSWORD=)
    • Run php artisan migrate
    • Run php artisan db:seed
  6. Start the server: php artisan serve
  7. Access at http://localhost:8000

Docker Setup (Edge Case)

Warning: Docker is provided as an edge case for PCs that cannot install PHP locally. Use XAMPP setup if possible.
  1. Install Docker Desktop and ensure it's running
  2. Clone the repository
  3. Run .\initial_start.bat (Windows) or docker compose up
  4. Access at http://localhost:8000
  5. Adminer available at http://localhost:8080 for database management

Common Pitfalls

  • XAMPP Path Detection: The setup script looks for XAMPP in common paths (C:\xampp, D:\xampp, C:\Program Files\xampp). If installed elsewhere, manual setup is required.
  • Database Creation: The dole_payroll database must exist before running migrations. The setup script handles this automatically.
  • .env Configuration: Ensure SESSION_DRIVER=database, CACHE_STORE=database, and QUEUE_CONNECTION=database are set for proper operation.
  • HRIS API: The HRIS API integration is currently stubbed with mock data. Configure HRIS_API_URL and HRIS_API_KEY in .env when the real API is available.

Key Modules

Regular Payroll

Flow:

  1. Payroll Officer creates batch (year, month, cutoff: 1st or 2nd)
  2. System generates PayrollBatch in draft status
  3. Payroll Officer clicks "Compute" → PayrollComputationService processes all active employees:
    • Fetches attendance from HRIS API (LWOP days, late minutes, undertime minutes, YTD gross)
    • Calculates gross income (basic + PERA + RATA, semi-monthly)
    • Calculates attendance deductions (LWOP, tardiness using Table IV)
    • Calculates government deductions (Pag-IBIG, PhilHealth, GSIS, WHT using annualized method)
    • Applies enrolled loan deductions
    • Computes net pay
  4. Batch status changes to computed
  5. Payroll Officer submits → pending_accountant
  6. Accountant certifies funds → pending_rd
  7. RD/ARD approves → released
  8. Cashier locks after disbursement → locked
  9. Force edit override allows reverting from locked to released (requires remarks)

Payslip Generation

Enhanced Features:

  • Payslips can be generated by month (combining both cutoffs) or by individual payroll batch
  • "My Payslip" feature allows employees to view their payslips via HRIS SSO
  • Signatory information is included on payslips for proper authorization
  • Payslip views have been enhanced with better formatting and layout

Special Payroll

Types: Newly Hired (pro-rated), Salary Differential, NOSI (Non-Optimization of Salary Increase), NOSA (Non-Optimization of Salary Adjustment)

Flow:

  1. HRMO creates special payroll batch with employee, effectivity dates, salary amounts
  2. Service computes pro-rated or differential amount based on working days
  3. Batch created in draft status
  4. Accountant approves → approved
  5. RD/ARD releases → released

TEV (Travel Expense Voucher)

Tracks: Cash Advance (requires liquidation) or Reimbursement

Flow:

  1. HRMO creates Office Order (authorization for travel)
  2. HRMO creates TEV linked to approved Office Order with itinerary lines
  3. TEV auto-submits on creation (no draft state — HRMO files directly)
  4. Accountant certifies → accountant_certified
  5. RD/ARD approves → rd_approved
  6. Cashier releases:
    • Cash Advance: Records cash advance amount, status → cashier_released
    • Reimbursement: Status → reimbursed
  7. After travel, HRMO certifies travel completion (optional Annex A expenses)
  8. For Cash Advance: Employee or HRMO files liquidation with actual amount spent
  9. System calculates balance due (positive = refund, negative = additional payment)
  10. Cashier approves liquidation → liquidated
Planned Enhancement: Future integration with HRIS to receive approved TEV requests via webhook instead of internal creation by HRMO. This would shift the workflow so that HRIS becomes the authoritative source for TEV requests, with approved TEVs being pushed to the Payroll System for processing. Timeline: TBD (pending HRIS coordination).

Employee Management

Flow:

  1. Employees are strictly sourced from HRIS API — no manual creation
  2. HRIS sync matches employees by employee_no, updates existing records, creates new ones
  3. Division matching uses division_code from API to local divisions
  4. Deduction enrollments track which loans/deductions apply per employee with effective dates
  5. Promotion history tracks salary changes over time
  6. Employees can now be linked to user accounts for HRIS SSO integration

User Management

Flow:

  1. Users can be assigned a main role and a sub role for flexible access control
  2. User accounts have active/inactive status for better account management
  3. Signatory management allows linking users to document signatory roles
  4. Users can be linked to employee records for HRIS integration
  5. Role-based access control is enforced throughout the system

Deduction Type Management

Flow:

  1. DeductionTypeController provides full CRUD operations for deduction types
  2. Deduction types can be created, edited, and deleted via CMS interface
  3. Deduction types are used in employee deduction enrollments
  4. Supports government deductions (GSIS, HDMF, PHIC, SSS) and loan deductions

Government Remittances

Flow:

  1. User selects year, month, cutoff
  2. System queries payroll entries for the period
  3. Deduction amounts are aggregated by deduction type
  4. Excel export generated with proper formatting for each agency (GSIS summary/detailed, HDMF 5-sheet, PHIC CSV, etc.)

The system integrates with an external HRIS (Human Resource Information System) for employee Single Sign-On (SSO) and attendance data synchronization.

Setup

  2. Install dependencies: npm install
  3. Start the HRIS server: npm start

Features

  • Employee Login: Login with Employee ID (EMP001-EMP082) and password (pass123)
  • JWT Generation: Generates signed JWT tokens for SSO authentication
  • Navigation: Provides links to Payroll and TEV systems
  • Dummy Data: Includes 82 employees with full profiles and attendance records

JWT Configuration

JWT configuration in HRIS/config.js:

  • Secret: dole-hris-payroll-shared-secret-2024
  • Expiration: 1 hour
  • Issuer: hris-system
  • Audience: payroll-system

Laravel Integration

The Laravel application receives JWT tokens at:

  • Payroll: http://localhost:8000/hris-auth?token={jwt}
  • TEV: http://localhost:8000/tev-hris-auth?token={jwt}

The JWT middleware (app/Http/Middleware/JwtAuth.php) validates the token and stores user data in session.

Production HRIS API

For production deployment, configure the following environment variables:

  • HRIS_API_URL — Base URL for the production HRIS API
  • HRIS_API_KEY — API key for HRIS authentication

Quirks & Decisions

HRIS API Stubbed

Critical: The HRIS API integration exists but returns mock data. Attendance data is currently stubbed and YTD (Year-To-Date) gross tracking is not implemented. This means withholding tax is under-computed for everyone.

The HrisApiService has real API calls but falls back to mock data when the API fails or returns errors. Attendance data (days_present, lwop_days, late_minutes, undertime_minutes) is stored in the attendance_snapshots table but is currently stubbed. YTD (Year-To-Date) gross — the cumulative gross income an employee has earned from January 1st up to the current payroll period — is critical for accurate withholding tax computation using the annualized method. Currently, YTD gross is hardcoded to 0 in the AttendanceSnapshot::toAttendanceArray() method (TODO at line 64), causing tax calculations to be incorrect. This must be implemented before production use.

TEV Auto-Submission

TEV requests are auto-submitted on creation. There is no draft state for TEVs. This is intentional because HRMO files TEVs on behalf of employees — the draft state was redundant. The submit() method exists but aborts with a 410 error if called.

Payroll Force Edit Override

Locked payroll batches can be force-edited back to released status by the Payroll Officer. This requires a mandatory remarks field (10-1000 characters) and is logged in the audit trail. This is a safety valve for post-disbursement corrections.

Attribute Aliases in Employee Model

The Employee model has attribute accessors that alias column names to match what services expect:

  • basic_monthly_salary → basic_salary column
  • pera_amount → pera column

This allows services to use consistent naming while the database uses shorter column names.

SpecialPayrollBatch Single Table

All special payroll types (newly_hired, salary_differential, nosi, nosa) share one table with a type column. This simplifies the schema but means some columns are only relevant for certain types (e.g., pro_rated_days only for newly_hired).

Table IV Minute Conversion

Tardiness and undertime minutes are converted to decimal days using a lookup table (DOLE Table IV), not a simple division. The TableIVConverter trait handles this. For example, 15 minutes = 0.031 days (from table), not 0.03125 computed.

Working Day Denominator Fixed at 22

All salary computations use a fixed 22 working days per month denominator, regardless of actual calendar days. This is per DOLE RO9 payroll rules and is hardcoded in multiple services.

Net Pay Verification

The /payroll/{payroll}/verify endpoint shows combined net pay for both cutoffs of a month, flags employees below ₱5,000 threshold, and identifies LBP loan deductions. This is for cashier verification before disbursement.

Deployment

No formal deployment scripts or CI/CD configuration found in the codebase. Deployment would be manual:

Manual Deployment Steps

  1. Clone repository to server
  2. Run composer install --no-dev --optimize-autoloader
  3. Copy .env.example to .env and configure production values
  4. Run php artisan key:generate
  5. Run php artisan migrate --force
  6. Run php artisan db:seed --force (if seeders are production-ready)
  7. Set APP_DEBUG=false in .env
  8. Set proper permissions for storage and bootstrap/cache
  9. Configure web server (Apache/Nginx) to point to public directory

Docker Deployment

The Docker setup is primarily for local development edge cases. For production Docker deployment:

  • Use Dockerfile and docker-compose.yaml as starting points
  • Add production environment variables
  • Configure persistent volumes for database and storage
  • Consider using a separate database container (currently uses external DB via Adminer)

Backups

Spatie Laravel Backup is configured. Use:

php artisan backup:run

Backup scripts (backup-data.bat, restore-data.bat) are available for Windows environments using Google Drive integration (not yet configured).
